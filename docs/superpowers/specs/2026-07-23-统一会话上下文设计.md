# 统一会话上下文设计

## 目标

在不改变现有文本和语音聊天记忆行为的前提下，以同一个 `conversationId` 统一管理文本、语音转写和待处理图片的短期上下文。

图片只在识图或参考图编辑意图下读取；普通文本聊天、工具调用和 DeepSeek 文本模型调用不自动携带图片。

## 现状与约束

- 文本和语音转写通过 `AiChatService.chat(conversationId, message)` 进入 Spring AI 的 `MessageChatMemoryAdvisor`。
- Spring AI 自动装配 `MessageWindowChatMemory` 与 `InMemoryChatMemoryRepository`，当前窗口为最近 20 条消息。
- iLink 文本、语音使用微信 `userId` 作为 `conversationId`；HTTP 接口仍使用现有默认值 `api-chat`。
- 图片由 `InMemoryImageContextService` 单独按用户缓存最近一张图片，固定 10 分钟过期，识图、编辑或普通文本回复后清除。
- DeepSeek 保持文本聊天模型；千问视觉和参考图生成仅在图片意图下使用。
- 不使用 iLink SDK 的 `ConversationContext` 作为业务上下文存储。该对象用于微信协议上下文令牌、输入状态和消息游标，不承载业务图片数据。

## 设计决策

新增 `ConversationContextService` 作为业务层唯一的会话上下文入口，以 `conversationId` 统一标识一段会话。

该服务不替换 Spring AI 的官方 `ChatMemory`：

- 文本与语音转写仍由现有 `AiChatService` 和 `MessageChatMemoryAdvisor` 自动存取。
- 图片作为会话附件上下文，由 `ConversationContextService` 管理，内部复用现有内存存储逻辑。
- 业务代码只能通过同一个 `conversationId` 访问文字记忆链路和图片附件链路。

不把原始图片直接写成 Spring AI `UserMessage.media`。虽然 Spring AI 支持多模态 `UserMessage`，但 `MessageChatMemoryAdvisor` 会把历史消息加入普通文本请求；这会使无图片意图的 DeepSeek 调用携带原图，违背本次需求。

## 组件调整

### ConversationContextService

新增接口及内存实现，职责仅为管理当前会话的待处理图片：

- `saveImage(String conversationId, byte[] imageBytes)`：保存或覆盖该会话的最新图片。
- `findActiveImage(String conversationId)`：取得未过期图片。
- `removeImage(String conversationId, PendingUserImage image)`：仅在仍为同一图片时删除，避免并发消息误删新图片。

实现继续使用 `ConcurrentHashMap<String, PendingUserImage>`、现有 MIME 识别逻辑和定时清理机制。图片有效期保持 10 分钟，不新增配置行为，不改动文本记忆的 20 条窗口。

`ImageContextService` 与 `InMemoryImageContextService` 被最小化替换为上述统一命名，以避免“图片上下文”和“会话上下文”存在两个业务入口。`PendingUserImage` 数据结构保持不变。

### iLink 消息接收

`IlinkMessagePollingService` 收到图片后调用：

`conversationContextService.saveImage(fromUserId, imageBytes)`

微信 `fromUserId` 即该会话唯一 ID，与随后文字、语音转写传入 `AiChatService.chat(userId, text)` 的 ID 保持一致。

### iLink 回复编排

`IlinkMessageReplyServiceImpl` 以 `userId` 查询图片附件：

1. 先获取 `findActiveImage(userId)`，供意图路由判断是否存在待处理图片。
2. 仅在 `IMAGE_UNDERSTAND` 或 `IMAGE_EDIT` 时把图片传给千问视觉或参考图生成服务。
3. 其他意图继续调用 `aiChatService.chat(userId, userText)`；不读取、不传递图片。
4. 图片完成识图、编辑或普通文本回复后，继续使用现有规则清除；过期图片由定时任务清除。

语音识别成功后已经调用 `aiChatService.chat(userId, voiceText)`，因此无需调整文本记忆逻辑。语音识别失败不写入任何聊天记忆或图片上下文。

## 数据流

```text
文字 / 语音转写
  -> conversationId（iLink 为 userId）
  -> AiChatService
  -> Spring AI ChatMemory（最近 20 条消息）

图片
  -> conversationId（iLink 为 userId）
  -> ConversationContextService（最新一张、10 分钟）
  -> 仅识图 / 编辑意图读取
  -> 千问视觉 / 参考图生成
```

## 非目标

- 不将原始图片自动加入 DeepSeek 的历史 Prompt。
- 不实现 Redis、JDBC 或其他持久化存储。
- 不改变 HTTP API 的默认 `api-chat` 会话行为。
- 不改变意图路由、模型选择、工具调用和现有图片处理规则。
- 不保留完整聊天历史；文本记忆仍由 Spring AI 的 20 条消息窗口控制。

## 验证计划

1. 单元测试：保存、读取、覆盖、移除和过期清理图片；不同 `conversationId` 相互隔离。
2. 单元测试：最新图片覆盖旧图时，处理旧图不会误删新图。
3. 现有图片上下文测试迁移至新服务名称后继续通过。
4. 回归验证：语音转写与文字继续使用同一个 `userId` 的 Spring AI 记忆。
5. 回归验证：普通文本请求不调用视觉模型、不读取图片字节；识图和编辑请求能读取同会话最新图片。
6. 构建验证：`mvn -q -DskipTests compile`。
